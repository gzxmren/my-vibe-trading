# Vibe-Trading & Gemini CLI 金融助手工作准则

本工作区已集成 Vibe-Trading 金融研究工具包。Gemini CLI 作为首席分析师，直接调用算法库和数据加载器执行任务。

## 📁 产出物存储与命名规范 (Mandatory)

所有分析生成的报告、数据、图表必须严格遵守以下规则：

1. **根目录**: 统一存放于项目根目录下的 `./reports/` 文件夹。
2. **文件夹命名**: `{股票名称}{股票代码}` (例如：`特变电工600089`)。
3. **文件命名格式**: `{股票市场}{股票代码}_{分析指令}_{时间}.{类型后缀}`
    - **股票市场**: SH (沪), SZ (深), HK (港), US (美), Crypto (币)
    - **时间格式**: YYYYMMDD
    - **分析指令**: 简短的中文/英文描述 (例如：`RBreaker回测`, `缠论分型`)
    - **示例**: `reports/特变电工600089/SH600089_RBreaker回测_20240513.md`

## 🌟 架构边界：Gemini-Native 分析体系 vs Vibe-Trading 底层

为了保证系统的可控性、稳定性和高扩展性，本工作区采取了明确的**“双层架构”**：

1. **底层组件与知识库 (Vibe-Trading Core & Skills)**：
   - **原材料库**：复用其优秀的数据加载器（`loaders` 下的 `mootdx`, `yfinance` 等）和高级数学算法库。
   - **知识蓝图库 (The 74 Skills)**：Vibe-Trading 提供的 74 个专业金融技能（已存放于 `./.gemini/skills/vibe-trading/`）是我们最宝贵的**算法蓝图与金融知识库**。这些技能包含了期权定价、宏观经济分析、基本面选股等无数深度的 Python 代码模板和业务逻辑。
   - **摒弃**：仅放弃使用其内置的、依赖外部 API Key 且极易报错的 `vibe-trading run` 命令行助手。
2. **核心执行层 (Gemini-Native Engines)**：
   - **定位**：我们建立的**“独立指挥中心”**。Gemini CLI 负责阅读并吸收底层 74 个技能的知识蓝图，将其转化为固化的高级分析引擎，存放于 `./scripts/gemini_engines/` 目录下。
   - **当前资产**：
     - `pro_report_gen.py`: 吸收了 RSI/MACD/SMC/缠论 等技能的联合诊断引擎。
     - `elliott_time_analyzer.py`: 吸收了波浪理论技能的时空矩阵引擎。

## 🛠️ 技术执行路径

1. **大脑吸收**: 当收到新类型的金融任务时，Gemini CLI 优先从 `./.gemini/skills/vibe-trading/` 中检索并学习相关技能的 Markdown 文档。
2. **工具铸造**: 学习完毕后，Gemini CLI 提取技能中的核心算法，在 `scripts/gemini_engines/` 下组装成具有自适应寻优能力、能生成 PDF 的高级自有引擎。
3. **感官调用**: 引擎运行过程中，调用 `agent/backtest/loaders/` 下的加载器获取极速原始数据。

## 📋 报告标准

每次分析必须同时产出：
- **Markdown 报告**: 包含核心结论、风险提示和策略建议。
- **CSV 数据**: 包含完整的交易流水和净值曲线。
- **(可选) 图像**: 如果涉及图表，保存为同名的 `.png`。

## 🧠 AI 深度思考与推理原则 (Mandatory)

在进行任何模块的集成、完善与重构时（如多因子选股、宏观分析等），**必须坚决摒弃“纯静态字符串模板拼接”的传统生成模式**。
所有的总结、研判与洞察，必须通过真实调用 LLM（大语言模型）进行深度推理生成：
1. **拒绝废话与静态模板**：研报不能是硬编码占位符的简单替换。必须让模型基于实时横截面数据、行业集中度、异动等真实情况进行逻辑归纳。
2. **结合实际业务逻辑**：不仅要罗列数据结果，必须通过 LLM 推理出个股间的隐性题材关联（例如“X只股票同属电力板块”），并给出针对性的操作建议与风险提示。
3. **全模块适用**：这不仅适用于多因子选股雷达，而是未来所有 Vibe-Trading 金融引擎开发的**第一准则**。
